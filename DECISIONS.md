# MultiConvert — Architecture Decisions

## API Selection

### Fiat Rates: frankfurter.app (not exchangerate.host)
**Reason:** `https://api.exchangerate.host/latest` now requires a paid subscription key (changed post-2023). `https://api.frankfurter.app/latest` is fully open-source (self-hosted by the community), free, no key required, and returns the same data format. Documents in README under API rate limits.

### Crypto Rates: CoinGecko free tier
**Reason:** As specified. The free tier at `/v3/simple/price` requires no key and covers all five required symbols. Rate limit is ~10–30 req/min depending on IP.

---

## Rate Storage Format

All rates are stored internally as **"units per USD"**:
- `ratesPerUSD["EUR"] = 0.926` → 1 USD = 0.926 EUR
- `ratesPerUSD["BTC"] ≈ 0.0000154` → 1 USD = 0.0000154 BTC

Conversion formula: `result = amount × ratesPerUSD[to] / ratesPerUSD[from]`

This choice:
- Avoids O(n²) fetch combinations
- Makes cross-rates (e.g., BTC → JPY) trivial
- Single cache entry covers all conversions

---

## Architecture: MV with @Observable

No MVVM. AppState is a plain `@Observable` class injected via SwiftUI environment. Views read state directly. Mutations go through AppState methods. This matches the spec exactly and avoids unnecessary ceremony.

---

## Persistence Strategy

| Data | Storage |
|------|---------|
| Recent currencies (MRU list) | `UserDefaults` as JSON-encoded `[String]` |
| Settings (decimal places, widget config) | `UserDefaults` via direct get/set |
| Rate cache | `FileManager` → `Caches/multiconvert_rates.json` |

Widget reads rates from the same Caches file — no App Group needed since both targets share the device's default `cachesDirectory`.

**Decision:** Skipped App Group for simplicity since the widget reads the JSON cache file directly from the shared Caches directory (same process space on simulator, same container on device for the main app). If App Group is required for distribution, the cache path in both `RateCache` and the widget's provider would change to `FileManager.default.containerURL(forSecurityApplicationGroupIdentifier:)`.

## Keypad Design

The keypad layout is 3-column (C / ⌫ / .) + (7 8 9) + (4 5 6) + (1 2 3) + (0), matching standard calculator ergonomics. Buttons are circular with a pressed-state animation via `DragGesture` (not `ButtonStyle` press state) to avoid SwiftUI's default 0.5s delay on highlight.

---

## Widget: StaticConfiguration (not AppIntentConfiguration)

**Reason:** `AppIntentConfiguration` requires an `AppIntent` for the user-selectable parameter. The spec says "user-pinned base amount" which is configured in Settings, not the widget gallery. `StaticConfiguration` is simpler and correct. The widget reads the base amount/currency from `UserDefaults` (set in Settings).

---

## Theme Colors

All hex values differ from Apple Calculator's palette:
- App background: `#1A1A1F` (dark charcoal, not Apple's `#1C1C1E`)
- Operator accent: `#C8872A` (amber-gold, not Apple's orange `#FF9F0A`)
- Digit buttons: `#2E2E38` (slightly purple-tinted dark, not `#333333`)
- Secondary text: `#9A9AA8` (cool gray, not `#8E8E93`)

---

## No App Icon Image

A placeholder `AppIcon.appiconset/Contents.json` is included with no image assets. The spec calls for a currency-glyph SVG/PNG. A designer should supply the final 1024×1024 PNG. The AccentColor is set to an amber tone matching the UI palette.

## IAP Placeholder

As specified, the IAP is present in `SettingsView.swift` as a grayed-out "Remove Ads (coming soon)" row with a `$1.99` price label. No StoreKit code is active. The structure is ready for activation:
1. Add StoreKit framework
2. Create `IAPManager` with `Product.products(for:)` and `.purchase()`
3. Wire the button to `IAPManager.purchase(productID: "com.jibaroenlaluna.multiconvert.removeads")`

---

## Xcode Project Generation

The `.xcodeproj` is generated by `gen_xcodeproj.py` (no Xcode, no xcodegen, no external tools). This means:
- The project file is reproducible and diffable
- To add a new source file: add it to the Python script lists and re-run `python3 gen_xcodeproj.py`
- All UUIDs are counter-based (sequential) for determinism

---

## Swift Testing vs XCTest

- Unit tests use Swift Testing (`@Suite`, `@Test`, `#expect`) — Xcode 15+ / Swift 5.9+
- UI tests use XCTest — required for `XCUIApplication`

---

## Offline Behavior

1. App launch: immediately shows cached rates (even if stale)
2. Background refresh via `.task { await state.refresh() }` on ContentView appear
3. Staleness threshold: 24 hours (matches the spec)
4. Stale indicator: amber warning bar with timestamp
5. Pull-to-refresh: `ScrollView` → `.refreshable` closure
